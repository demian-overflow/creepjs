Why this exists. Upstream creepjs reads identity attributes from window.navigator in the page context. A browser that spoofs only the page-side navigator (most fingerprint-evasion frameworks) passes those checks. Real-world detectors at Telegram / Cloudflare / Akamai-grade do not stop there — they cross-check the same attribute against a Worker context, an iframe, and the request headers their server actually receives. Any divergence is a strong "this UA is lying" signal.

This page reads navigator-derived identity from five contexts and diffs them against each other and against /echo (an openresty endpoint on this host that returns the headers the server saw). Per-row red means that attribute disagrees somewhere — your shim missed a context. See nightglow/docs/defects/fingerprint-gaps.md for the rationale and what to do about each leak.

Modern Chrome API surface

Existence checks for APIs Chrome 90+ exposes. A "Chrome 124" UA that returns undefined for any of these is suspicious to detectors that probe the version-coherent surface. Doesn't fail per se — informational.

Timer + stack format

High-precision performance.now() in a cross-origin context, or an async stack trace whose frames don't match Chrome's format, is a tell that something below the JS surface is wrong.

Wire-level TLS ClientHello

Fetched from https://tlsprobe.orderout.noogoo.ch:8443/ — a self-hosted Go service that peeks the TLS ClientHello before the handshake completes and returns the parsed shape. JS has no direct access to this layer, so this is the only way to surface what real-world bot detectors at Cloudflare / Akamai / Telegram-grade actually use to flag spoofed UAs (JA3, JA4, GREASE presence, supported_groups). Compared against a baseline captured from a real Chrome 124. Red = differs from Chrome 124 in a way that's a known tell.
